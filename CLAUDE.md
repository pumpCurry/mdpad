## 📌 リポジトリ構造・バージョン参照の原則（REQUIRED）

### 0-1. フォルダ構成の正（Single Source of Truth）
- リポジトリのフォルダ構成・モジュール関係・ドキュメントの導線は、原則として **`/README.md` およびそのリンク先の `.md` 群**から確認すること（MUST）。
- この AGENTS.md 内でフォルダ構成を決め打ちしない。必要な場合は `/README.md` を根拠として明記すること（MUST）。

### 0-2. 現行 version の正（Single Source of Truth）
- **現在のバージョン指定（現行の version 値）**は、原則として **`/README.md`** から確認すること（MUST）。
- `/README.md` にバージョン更新規則（例：第4ブロック増分など）がある場合は、それを最優先で適用する（MUST）。
- 上記規則に従って算出した **新しいバージョンは `/README.md` 自体にも反映**し、各実行ごとにインクリメントされる状態を維持すること（MUST）。
- ファイル内の `@version` / `Version:` などの書式は **既存ファイルの慣例を優先**し、その値（現行バージョン）を `/README.md` に合わせて更新すること（MUST）。

### 0-3. monorepo 前提（core / system / docs）
- `core/` 配下に `system/{systemName}/` が存在しうる。
- 各システムのルート（`core/system/{systemName}/`）配下には、それぞれの `readme.md` や `docs/` 等の `.md` が存在しうる（MUST認識）。
- 各プログラムは **各システムのルートを基準に相対パスで動作する**ように記載・実装する（MUST）。
  - 絶対パス前提（例：`/home/...`、`C:\...` 等）の設計・実装は禁止（MUST NOT）。

### 0-4. 更新履歴の正（updates.md / updates_template.md）（REQUIRED）
- 更新履歴は `./updates.md` に記帳する（MUST）。
- 記帳フォーマット（規格）は `./updates_template.md` を正とする（MUST）。
- `updates.md` の先頭には必ず次のヘッダを置く（MUST）。以後、更新エントリは **ヘッダ直後に先頭追記**で挿入され続ける（MUST）。

  ```
  # 更新履歴(updates)
  - 現在のバージョン: ....
  - 最終更新日時: ...
  ```

- `updates.md` ヘッダは更新のたびに更新する（MUST）
  - `現在のバージョン`：更新後の `/README.md` の値と一致させる（MUST）
  - `最終更新日時`：先頭追記した最新エントリの `date_jst` と一致させる（MUST）
  - まだエントリが存在しない場合は `未記入` としてよい（MAY）

## 🔁 ワークフロー・オーケストレーション（REQUIRED）

AIエージェントがタスクを遂行する際の行動原則を定める。
コード出力ルール（後述）と合わせて、作業プロセス全体を統制する。

### 1. Plan Mode Default（計画モード優先）（MUST）
- **非自明なタスク（3ステップ以上、またはアーキテクチャ判断を伴うもの）には必ず計画モードに入る**（MUST）。
- 問題が発生した場合は、即座に **停止して再計画** する。押し通さない（MUST）。
- 計画モードは構築だけでなく、**検証ステップにも使う**（SHOULD）。
- 曖昧さを減らすために、**詳細な仕様を事前に書き出す**（SHOULD）。

### 2. サブエージェント戦略（SHOULD）
- メインのコンテキストウィンドウをクリーンに保つため、**サブエージェントを積極的に活用する**（SHOULD）。
- リサーチ、探索、並行分析はサブエージェントに委譲する（SHOULD）。
- 複雑な問題にはサブエージェントを増やして計算資源を投入する（MAY）。
- **1タスク1サブエージェント**で集中的に実行する（SHOULD）。

### 3. 自己改善ループ（MUST）
- ユーザーから修正指示を受けた場合、**`tasks/lessons.md` にパターンとして記録する**（MUST）。
- 同じミスを防止するルールを自分で記述する（MUST）。
- これらのレッスンを繰り返し改善し、ミス率を下げる（SHOULD）。
- セッション開始時に、該当プロジェクトのレッスンをレビューする（SHOULD）。

### 4. 完了前検証（Verification Before Done）（MUST）
- **動作を証明せずにタスク完了とマークしてはならない**（MUST）。
- 必要に応じて main ブランチと変更のdiffを確認する（SHOULD）。
- 「スタッフエンジニアがこれを承認するか？」と自問する（SHOULD）。
- テストを実行し、ログを確認し、正しさを実証する（MUST）。

### 5. エレガンスの追求（バランス重視）（SHOULD）
- 非自明な変更では、一度立ち止まって「もっとエレガントな方法はないか？」と問う（SHOULD）。
- ハッキーだと感じたら「今知っていることすべてを踏まえて、エレガントな解を実装する」（SHOULD）。
- **単純で明白な修正には適用しない** — 過剰設計を避ける（MUST NOT：過剰設計）。
- 提示する前に、自分の成果物にチャレンジする（SHOULD）。

### 6. 自律的バグ修正（MUST）
- バグ報告を受けたら、**手取り足取りを求めず、自分で修正する**（MUST）。
- ログ、エラー、失敗テストを指摘し、それらを解決する（MUST）。
- ユーザー側のコンテキストスイッチをゼロにする（SHOULD）。
- CIテストが失敗していたら、指示を待たずに修正に向かう（SHOULD）。

## 📋 タスク管理（REQUIRED）

1. **Plan First**：`tasks/todo.md` にチェック可能な項目で計画を書き出す（MUST）。
2. **Verify Plan**：実装開始前にユーザーに計画を確認してもらう（SHOULD）。
3. **Track Progress**：作業進行に応じて項目を完了にマークする（MUST）。
4. **Explain Changes**：各ステップでハイレベルなサマリを提供する（MUST）。
5. **Document Results**：`tasks/todo.md` にレビューセクションを追記する（SHOULD）。
6. **Capture Lessons**：修正を受けた場合は `tasks/lessons.md` を更新する（MUST）。

## 🎯 コア原則（REQUIRED）

- **Simplicity First（単純さ優先）**：あらゆる変更をできるだけシンプルにする。影響範囲を最小限にする（MUST）。
- **No Laziness（怠惰の排除）**：根本原因を探る。一時的な修正をしない。シニア開発者の基準で取り組む（MUST）。
- **Minimal Impact（最小影響）**：必要なものだけに触れる。バグを生まない変更を心がける（MUST）。

## ✅ updates.md 記帳の実務手順（順序固定）（MUST）

更新（仕様・実装・ドキュメントのいずれか）を行う場合、次の順序を必ず守ること（MUST）。

1. `/README.md` を読み、現行 version と更新規則を確認する（MUST）。
2. `/README.md` の規則に従い、新 version を確定し、`/README.md` に反映する（MUST）。
3. 変更した各ファイルについて、先頭ヘッダの `revision` を更新する（MUST）。
   - 既存に `revision` が無ければ `1`
   - 既存に `revision` があれば必ず `+1`
4. 変更した各ファイルについて、`@version`/`Version:` を新 version に更新し、`@lastModified`/`LastModified:` を JST で更新する（MUST）。
   - `@since`/`Since:` が既にある場合は保持し、書き換えない（MUST）
5. `./updates.md` のヘッダを更新する（MUST）。
   - `現在のバージョン`：新 version
   - `最終更新日時`：これから追記する最新エントリの `date_jst`
6. `./updates_template.md` に従い、**`updates.md` のヘッダ直後へ 1エントリを先頭追記**する（MUST）。
7. 構文エラーや参照崩れが無いことを最低限確認する（MUST、後述「コミット前の最低テスト」）。

## ✅ 絶対遵守事項（REQUIRED）

### 1. 省略禁止（MUST NOT）
- 改造や改修に関係しない部分を省略してはならない。
- `function` / `class` / モジュールの一部だけを書くことは禁止。
- **前後の依存コードを含めてすべて提示すること。**
- 例外：リポジトリ都合でファイル全量が物理的に提示不能な場合は、対象範囲を明示し、欠落範囲と理由を明記すること（ただし原則禁止）。

### 2. ドキュメントコメント必須（MUST）
出力するコードには、言語ごとに以下のドキュメント形式を **必ず** 付与する。

- **JavaScript**：詳細な **JSDoc**（引数・戻り値・処理内容・型）
- **Python**：PEP257準拠の **docstring** + **型ヒント**
- **HTML**：先頭コメントで **責務/構造/主要ID・class/依存** を説明
- **CSS**：先頭コメントで **責務/適用範囲/依存class/テーマ変数/副作用** を説明

### 3. 日本語による詳細コメント（MUST）
- 目的、分岐の理由、背景、注意点を日本語で丁寧に記述すること。
- 「なぜそうするか」が読み取れない実装は無効。

### 4. 順序維持（SHOULD）＋条件付き再配置（MAY）
- **原則（SHOULD）**：関数・クラス・宣言・構造の順番は元のコードに準じ、変更しない。
- **例外（MAY）**：以下の合理的理由があり、影響を説明できる場合に限り再配置を許可する。
  - 手続き・宣言の順序が動作に影響する（初期化順、依存解決、import順など）
  - 機能単位でまとめることで保守性が大きく上がる（同カテゴリ関数の集約、責務の分離）
- **説明義務（MUST）**：再配置した場合は「どこを」「なぜ」「影響は何か」を
  - (a) `updates.md` の当該エントリ、または
  - (b) コードコメント（該当箇所）
  のいずれかに明記すること。

### 5. 不要な削除・短縮の禁止（MUST NOT）
- 変更がない部分であっても、明示されない限り省略・削除してはならない。
- 「ついで整理」「未使用っぽいので削除」は禁止（削除は明示指示がある場合のみ）。

### 6. 完全動作を前提とする（REQUIRED）
- HTML/CSS/設定ファイル/静的資産（参照されるもの）も含めて、**全体として動作する状態**で提示すること。
- 各システムは **そのシステムルートを基準に相対パスで解決できる**こと（MUST）。
- 部分コード提示のみで動作不能になる出力は無効。

### 7. ファイル先頭ヘッダ（REQUIRED）
対象ファイルには、言語に応じた **先頭ヘッダ**を入れること。

- **revision（リビジョン）は全ファイル必須**（MUST）
  - 編集元のファイルに `revision` が無い場合：`1` を付与する
  - 編集元のファイルに `revision` がある場合：**必ず +1** する
- `@since` / `Since:` は、既に記載がある場合は保持し、書き換えない（MUST）
- `@version` / `Version:` は更新する（MUST）
  - 現行 version の値は `/README.md` を正として更新する（MUST）
- `@lastModified` / `LastModified:` は `YYYY-MM-DD hh:mm:ss (JST)` 形式で更新する（MUST）
- `@todo` / `Todo:` は今回達成できなかった項目があれば箇条書きで追記する（MUST）

> 書式（タグ名や行の並び）は、既存ファイルの慣例を優先する。

#### 7.1 先頭ヘッダ（汎用テンプレート）

**JavaScript（JSDoc）**
```js
/**
 * @fileoverview
 * @description {概要}
 * @file {ファイル名}
 * @module {モジュール名}
 * @version {README.mdの現行versionに合わせて更新}
 * @since {既存があれば保持}
 * @revision {無ければ1 / あれば+1}
 * @lastModified YYYY-MM-DD hh:mm:ss (JST)
 * @todo
 * - {未達項目があれば}
 */
```

**Python（モジュールdocstring）**
```py
"""
{概要}

Version: {README.mdの現行versionに合わせて更新}

Revision: {無ければ1 / あれば+1}
LastModified: YYYY-MM-DD hh:mm:ss (JST)
Todo:
- {未達項目があれば}
"""
```

**HTML**
```html
<!--
{概要}

- {未達項目があれば}

```

**CSS**

## ⚠ 禁止事項（DO NOT）
- canvas機能の使用は禁止。
- 三点リーダー等（`...` / `…`）や「省略」等の省略表記による省略は禁止。
- 「省略しても動きます」という注釈は禁止。
- ドキュメントコメント無しの出力は禁止。
- 依存関係を壊す「ついで変更」（命名変更、不要な整形、未指示の最適化）は禁止。
- **絶対パス前提の実装は禁止**（例：`/home/...`、`C:\...`、固定ドライブ文字、固定ユーザ名 等）。
- **updates.md を更新せずに、仕様/実装/ドキュメント更新を出力することは禁止**。
- **動作を証明せずにタスクを完了とマークすることは禁止**（ワークフロー§4参照）。
- **一時的な修正（ワークアラウンド）で根本原因を放置することは禁止**（コア原則「No Laziness」参照）。

## 📘 言語別ドキュメント規約（REQUIRED）

### A) JavaScript（JSDoc）
- すべての関数（公開/内部）にJSDocを付与する（MUST）。
- `@param` / `@returns` / `@throws`（該当時）を省略しない（MUST）。
- 型は `{string}` `{number}` `{Object}` などを明示する（MUST）。

テンプレ：
```js
/**
 * {関数の要約}。
 *
 * 【詳細説明】
 * - {動作フロー}
 * - {注意点}
 *
 * @function {関数名}
 * @param {型} name - 説明
 * @returns {型} - 戻り値説明
 * @throws {Error} - 例外条件（あれば）
 * @example
 * // 使用例
 * const v = func(x);
 */
```

### B) Python（docstring + 型ヒント）
- すべての関数/メソッド/クラスに docstring を付与（MUST）。
- すべての引数/戻り値に型ヒントを付与（MUST）。
- 例外を投げる場合は docstring に明記（MUST）。

テンプレ：
```py
def func(x: int, y: int) -> int:
    """
    {関数の要約}。

    詳細:
        - {動作フロー}
        - {注意点}

    Args:
        x (int): 説明
        y (int): 説明

    Returns:
        int: 戻り値の説明

    Raises:
        ValueError: 例外条件（あれば）

    Examples:
        >>> func(1, 2)
        3
    """
    return x + y
```

### C) HTML（構造と依存の説明）
- 先頭コメントに以下を必ず書く（MUST）。
  - このHTMLの責務（何の画面/部品か）
  - 主要なDOM構造の意図（セクション分割）
  - 参照するJS/CSS、主要ID/class、data-* 属性の意味
  - 変更時に壊れやすい点（イベント紐付け等）

### D) CSS（責務/範囲/依存/副作用）
- 先頭コメントに以下を必ず書く（MUST）。
  - このCSSの責務（テーマ、コンポーネント、ユーティリティ等）
  - 適用範囲（ページ全体/特定コンポーネント）
  - 依存class/変数（CSS変数、テーマ色、外部ライブラリ前提）
  - 副作用（他要素に波及する指定）と回避策

## ✅ CI / 静的チェック（推奨、ただし最低限は必須）
- **必須（MUST）**：出力した各ファイルが構文エラーを起こさないこと。
- **推奨（SHOULD）**：各言語で一般的な lint / format / typecheck を導入して維持する。
  - JS: ESLint（+ JSDocチェック）
  - Python: ruff/flake8 + mypy/pyright（必要に応じて）
  - CSS: stylelint
  - HTML: HTML lint / ビルド検証（構成に応じて）

## ✅ コミット前の最低テスト（MUST）

- **共通**
  - 構文エラーが無い
  - 追加した変数/定数/構造体/ID/class に二重定義・定義漏れが無い
  - 参照パス（import / script / link / asset）が破綻していない
  - 相対パス前提が守られている（絶対パスが混入していない）

- **Python**
  - 少なくとも import / 実行時に即死しない（例：`python -m py_compile` 相当）

- **JS/HTML/CSS**
  - ブラウザで読み込んだ瞬間に致命エラー（未定義変数、要素ID不一致等）が出ない
  - JSが参照するDOM要素（id/class/data-*）がHTML側に存在する

## 🏗 ビルド・リリース規則（MUST）

### B-1. バージョン番号体系

- アプリのリリースバージョンは **`v{major}.{minor}.{buildNumber(5桁ゼロ埋め)}`** 形式（例: `v1.1.00063`）。
- **ビルド番号（buildNumber）= master ブランチの `git rev-list --count` + 1**（MUST）。
  - つまり **main（master）ブランチのコミット総数 + 1** がビルド番号となる。
  - `scripts/update-build-number.js` が自動算出し、`build-number.json` と `package.json` の `version` フィールドに書き込む。

### B-2. ブランチでのビルド時の注意（MUST）

- **feature ブランチや worktree で作業中は、`git rev-list --count HEAD` が master より少なくなる場合がある**。
  - ブランチを切った時点のコミット数が master より古いと、バージョンが逆進する。
  - `update-build-number.js` はこの問題を自動補正する（master のコミット数 + ブランチ固有コミット + 1 を使用）。
- **ビルド前に必ず以下を確認すること**（MUST）：
  1. `git rev-list --count master` を実行し、master のコミット数を確認する。
  2. GitHub の最新リリースタグ（例: `v1.1.00062`）と比較し、ビルド番号がそれ以下にならないことを確認する。
  3. もし逆進が発生する場合は、**ビルドを中止し、原因を調査してから再実行する**（MUST）。

### B-3. インストーラバージョンの一致（MUST）

- **NSIS インストーラや EXE に埋め込まれるバージョンは、アプリ内表示バージョンと完全に一致しなければならない**（MUST）。
  - `package.json` の `version` フィールドがインストーラのファイル名・メタデータに使われる。
  - `update-build-number.js` が `package.json` の `version` を `{major}.{minor}.{buildNumber(5桁ゼロ埋め)}` 形式に自動更新する。
  - **`package.json` の `version` が `1.1.0` のような不完全な値のままビルドすることは禁止**（MUST NOT）。
  - ビルド後、インストーラのファイル名（例: `mdpad-1.1.00063-setup.exe`）がアプリ表示バージョン（例: `v1.1.00063`）と一致することを目視確認する（MUST）。

### B-4. ビルド前チェックリスト（MUST）

ビルド実行前に以下を順に確認すること：

1. `git rev-list --count master` でコミット数を確認
2. GitHub の最新リリースタグと比較し、バージョン逆進がないことを確認
3. `npm run build:renderer` を実行（webpack/esbuild バンドル）
4. ビルド後の `build-number.json` と `package.json` の `version` が正しい値になっていることを確認
5. `npx electron-builder` 実行後、出力されたインストーラのファイル名にバージョンが正しく含まれることを確認
6. スモークテスト（`npm run test:smoke`）を実行し、リグレッションがないことを確認
